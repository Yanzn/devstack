# 设计哲学

> [English version](philosophy.md)

devstack 建立在五条信念上。如果你不认同其中任何一条，这个插件会和你对着干。

## 1. 好代码来自流程

AI 智能体默认走最短路径，产出看似合理的答案。这条路径跳过规格、跳过测试、跳过 review，产出的代码在 demo 里能跑、在生产里会炸。devstack 用强制性工作流把"长路径"做成比"短路径"更便宜的选择。

这条信念来自 [Superpowers](https://github.com/obra/superpowers)。在 `flow/` 和 `core/` 层里不可协商。

## 2. 好代码靠规范维持

只有流程纪律、没有内容知识，产出的是测试通过的平庸代码。devstack 的 `standards/` 层教智能体"什么是好代码"——覆盖 API、UI、安全、性能、文档、部署——汲取自业内公认的最佳实践（OWASP Top 10、Core Web Vitals、Hyrum's Law、trunk-based 开发、Google 工程实践）。

这条信念来自 [agent-skills](https://github.com/addyosmani/agent-skills)。

## 3. 流程和规范正交，要叠加使用

两个上游项目各解决了一半问题。流程层告诉你**什么时候做什么**，规范层告诉你**把事情做好的标准**。devstack 的核心论点是：两者都需要，合适的抽象是三层互不知晓对方实现的堆叠：

```
┌─────────────────────────────────────────────────┐
│  flow/      brainstorm → plan → work → review   │  ← 工作流骨架
├─────────────────────────────────────────────────┤
│  core/      TDD · debug · verify · increment    │  ← 跨切纪律
├─────────────────────────────────────────────────┤
│  standards/ API · security · UI · perf · ship   │  ← 领域知识
└─────────────────────────────────────────────────┘
```

换掉一层，其他两层照常工作。在 `standards/` 下加 iOS skill 不动 `flow/`。重写 `flow/` 不动 `standards/`。

## 4. 小任务不需要仪式，大任务必须讲究

- **小**：改个 typo、重命名变量、调整配置。`core/` 层自动生效（加行为就 TDD，声明完成前要验证）。除此之外什么都不触发——没有 brainstorm、没有 plan、没有正式 review。
- **中**：一个涉及几个文件的功能。从 `/plan` 开始（如果需求已经清楚就跳过 brainstorm）。
- **大**：新功能、新子系统、跨模块或跨团队的工作。完整流程：`/brainstorm` → `/plan` → `/work` → `/review` → `/ship`。

入口由你选。Skill 不会强加超出任务规模的仪式，但一旦进入某个阶段，就严格要求按标准完成。

## 5. 证据胜过声明

每个 skill 都以验证闸门结尾。"看起来对"不算完成——测试通过、干净的构建日志、或运行时数据才算。这既对智能体的产出适用（它们必须证明自己的工作），也对 devstack 本身适用（每个 skill 都有可检查的退出条件）。

这条信念两个上游项目共有。

---

## 落到实处意味着什么

- 你输入 `/brainstorm` 时，智能体**不会**让你跳到写代码。一刀切。
- 你开始写实现代码但还没写失败测试，TDD skill 会让你删掉代码重写。
- 你声称某功能完成但没跑验证，智能体会追问证据并等你补上。
- 你要做一个 API，API-design skill 会逼你先定契约、想清楚错误语义，哪怕你只是"原型"。

这是有意的。如果你想要一个永远说对话的温顺智能体，请不要用 devstack。

## 归属可追溯设计

每个 devstack skill 顶部都有 HTML 注释注明上游出处（见 [CREDITS.md](../CREDITS.md)）。这意味着：

- 当 Superpowers 或 agent-skills 更新时，你可以对比它们的新版本和我们 skill 里注明的出处版本，决定要不要拉进来
- 当你读某个 skill 想深入了解时，注释指向源仓库可以追根溯源
- 当你扩展 devstack 加新 skill（比如 iOS 开发），标为 `[NEW]` 或指向新的上游来源，保持同样的追溯纪律
